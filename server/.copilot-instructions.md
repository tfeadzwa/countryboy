# Copilot Instructions for the Countryboy Project

These instructions help Copilot understand the architecture, coding conventions, and best practices already implemented in the Countryboy bus ticketing system. When generating or editing code, Copilot should reference this document to maintain consistency.

## 🧱 Project Structure

- `server/` contains the Express + TypeScript backend.
  - `src/controllers` – route handlers.
  - `src/services` – business logic interacting with Prisma.
  - `src/repositories` – (if added) low-level DB wrappers.
  - `src/middleware` – auth, RBAC, validation, error handling, request id, device auth.
  - `src/utils` – logger and other utilities.
  - `src/validators` – Zod schemas for request validation.

- `prisma/` – schema definitions and migrations.
- `frontend/` – React admin UI (mock data currently).
- `mobile/` – Flutter conductor app scaffold.
- `PROJECT_DOCS.md` – living documentation of system design and conventions.

## 📌 Coding & Architectural Conventions

- **Use TypeScript.** Strict mode is enabled. All new modules must be typed.
- **Paths:** avoid using `@/` aliases; use relative imports `../` to simplify runtime resolution.
- **Validation:** every route must validate inputs using Zod schemas defined under
  `src/validators/schemas.ts` and the `validate` middleware.
- **Error handling:** throw errors or pass them to `next()`; the centralized
  error middleware formats responses with `code`, `message`, `details`, and
  `requestId`.  Prisma unique‑constraint errors (`P2002`) are translated to
  409 responses with a friendly message via `formatPrismaError`.  Callers may
  pass the request payload as a second argument; the helper will insert the
  actual conflicting value(s) into the message and the middleware will echo
  back the attempted payload under `attempted` so the client can re-populate
  its form.
- **Logging:** use the `logger` from `src/utils/logger.ts`. Log structured JSON in
  production; include context (requestId, userId, depotId, deviceId, durationMs).
- **Transactions:** wrap multi-step operations in `prisma.$transaction`. This is
  essential for ticket issuance, serial allocations, and any update that spans
  multiple tables.
- **Security:**
  - Use `helmet` and configure CORS via `CORS_ORIGINS`.
  - Rate-limit sensitive endpoints (e.g. login).
  - Validate device tokens for sync; store per-device secrets in `tblDevices.token`.
  - Protect against SQL injection by using Prisma's query builder.
- **Request context:** attach `requestId` middleware; propagate user and depot
  information via `req.user` and `req.depotId` (populated by auth middleware).
- **Database schema:** prefix tables with `tbl`; include `depot_id`, audit fields,
  and create appropriate indexes (e.g. on `tblTickets`).
- **Migrations:** use `npx prisma migrate dev` to create and apply migrations.
  Keep the schema synchronized.

## 🔧 Features Already Implemented

- Authentication with JWT, roles (SUPER_ADMIN, DEPOT_ADMIN).
- RBAC middleware (`requireRole`).
- Depot scoping middleware.
- Zod validation for all endpoints.
- Structured logging (morgan + winston) with custom tokens.
- Rate limiter on login.
- Device authentication for sync endpoints.
- Sync endpoints (`/sync/push`, `/sync/pull`) with idempotent upserts.
- CRUD support for depots, agents, devices, fleets, routes, fares, trips, tickets.
- Metrics endpoints and simple overview service.
- Error handler that formats JSON and hides stack in production.

## ✅ When adding new functionality

1. **Define a schema** in `src/validators/schemas.ts` covering body/params/query.
2. **Add a service** function for business logic; ensure Prisma operations are scoped
   to `depotId` and wrapped in transactions when necessary.
3. **Add a controller** that calls the service and handles request/response, using
   `AuthenticatedRequest` where needed.
4. **Create a route** file importing the controller, middleware, and validation
   schemas; wire it into `src/index.ts`.
5. **Update documentation** (`PROJECT_DOCS.md`) with descriptions of the endpoint
   and any new architectural notes.
- **Write tests** (Jest + supertest) for new services and routes, mocking
  Prisma or using a test database.  Integration tests should exercise error
  responses (e.g. duplicate data returns 409) to prevent regressions.

## 🧾 Documentation & Testing

- Add notes to `PROJECT_DOCS.md` for every architectural change or new feature.
- Use Postman collections for manual API testing; document examples in the docs.
- Maintain `.copilot-instructions.md` with updated architecture guidelines.

Following these instructions will keep the backend consistent, secure, and
maintainable as the system evolves.
